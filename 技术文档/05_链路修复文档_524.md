## 二、修复详情

### 2.1 禁止硬编码关系类型（`rag_agent.py:248-253`）

**问题**：LLM 生成的 Cypher 如 `MATCH ()-[r:儿子]->()` 或 `-[r:妻子]->`，因图谱缺乏标准化本体，命中率恒为 0。

**修复**：在 `cypher_prompt` 的「绝对强制法则」中加入**法则零**：

```
🛑 法则零：禁止硬编码关系类型 (No Hardcoded RelTypes)
   - 绝对禁止在 Cypher 中指定具体的 relationship type
   - 必须使用无向且无类型的泛匹配 -[r]-
   - 正确示例：MATCH (a:`人物`)-[r*1..2]-(b:`人物`) WHERE a.name CONTAINS "王夫之"
   - 让下游回答大模型根据返回的 type(r) 和 r.detail 做逻辑推理
```

### 2.2 禁止弃用的 `id()` 函数（`rag_agent.py:255-258`）

**问题**：LLM 使用 Neo4j 5.x 已弃用的 `id()` 函数，触发 `01N02` 警告。

**修复**：在「绝对强制法则」中加入**法则一**：

```
🛑 法则一：禁止使用弃用的 id() 函数 (Neo4j 5.x 语法)
   - 当前为 Neo4j 5.x，id() 已被彻底弃用
   - 判断节点不同：WHERE a <> b（直接比较节点本身）
   - 绝对不允许 id(a) <> id(b) 或任何含 id() 的写法
```

### 2.3 独立实体直接查询法则（`rag_agent.py:259-263`）

**问题**：用户问"气一元论是什么"时，LLM 生成 `MATCH (a)-[r]-(b) WHERE a.name CONTAINS "王夫之" AND b.name CONTAINS "气一元论"`，但「气一元论」是孤立节点，查询返回空。

**修复**：在「绝对强制法则」中加入**法则二**：

```
🛑 法则二：独立实体直接查询法则 (Direct Entity Match for Concepts)
   - 当询问具体概念/思想/作品/事件时，直接 MATCH 该节点
   - ✅ MATCH (n) WHERE n.name CONTAINS "气一元论" RETURN n.name, labels(n)[0], n.description
   - ❌ 不要强制要求与主语有人物连线
```

### 2.4 扩充 JSON Schema 示例防止长文本丢失（`extractor.py:46-63`）

**问题**：原示例中「气一元论」description 只有一句话，模型学习了"概念只需一句话"的模式，导致数百字的哲学释义被丢弃。

**修复**：将 `JSON_SCHEMA_DEFINITION` 示例全面扩充：

| 实体 | 原 description 长度 | 修复后 |
|------|-------------------|--------|
| 王夫之 | ~30 字 | ~120 字（完整生平简介） |
| 气一元论 | ~30 字 | ~150 字（核心哲学论证全文） |
| 理势合一 | 不存在 | **新增**，~130 字详细哲学解释 |
| 曾国藩评价 detail | ~20 字 | ~80 字（完整原文摘抄） |

关键策略：**示范教学（Show, Don't Tell）**，让模型通过高质量示例理解 `description` 必须是完整的原文摘抄，而非一句话概括。

### 2.5 防崩溃严打法则（`extractor.py:19-23, 76-79`）

**问题**：大模型将长句子甚至整段话塞入 `name` 字段；将毫不相干的事件与哲学概念强行拉线。

**修复**（两层防护）：

**Pydantic 层** — `Entity.name` 加物理约束：

```python
name: str = Field(
    ...,
    description="实体名称，必须极度精简（通常2到10个字）。绝对禁止将长句子、完整段落或超过15个字的内容塞进name！",
    max_length=50,  # 物理截断锁
)
```

**System Prompt 层** — 新增第 5 条规则：

```
5. 🛑 绝对红线 — 防崩溃严打法则（违反即作废）：
   - name 必须是专有名词，极度精简（2-10字），超过15字就是错误
   - 所有长篇解释只能塞进 description 或 detail 字段
   - 严禁将毫不相干的事件与哲学概念强行建立连线
```

## 四、核心设计思想

1. **分层防御**：Prompt 指令（软约束）+ Pydantic Schema（硬校验）双重保障，即使 LLM 犯错也会在解析层被拦截。
2. **示范优于描述**：通过 `JSON_SCHEMA_DEFINITION` 提供"好答案长什么样"，比反复强调"description 要详细"更有效。
3. **泛匹配优于精确匹配**：Cypher 层不做语义判断，只做宽泛的数据捞取，将逻辑推理交给下游回答大模型。
