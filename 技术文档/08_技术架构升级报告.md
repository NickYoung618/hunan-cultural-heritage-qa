# 技术与架构升级汇报：向量数据库与AI应用——语义搜索

## 一、痛点：单一图数据库的语义天花板

在过往的研发中，我们的湖湘文化 GraphRAG 系统以 Neo4j 图数据库为核心，通过 Text-to-Cypher 技术，将用户自然语言翻译为图查询语句，精准命中人物、地点、事件等结构化实体关系。例如，当用户问"王夫之和曾国藩是什么关系"时，系统可以在图谱中检索出一条明确的二度关系链路。

然而，当我们面对更深层的文化精神类提问时，图数据库便暴露出其固有的局限。试问：**"什么是湖湘文化中'吃得苦、霸得蛮'的精神？"** 或 **"岳麓书院的经世致用学风如何影响了近现代湖南？"**

这类问题的答案，不对应图谱中任何一个具体的节点（Node）或关系（Edge）。它是一种**宏观语义**，散布在历史文献摘录、人物评价、事件背景描述的自由文本之中。Neo4j 的 Cypher 查询依赖精确的结构化模式匹配（Pattern Matching），无法处理这种基于"语义相似度"的模糊检索。换言之，**图数据库能回答"谁认识谁"，但无法回答"一种精神是什么"**。这是单体图谱检索架构的核心痛点。

## 二、创新：对标的 ACM MM 2025 顶会 HM-RAG 三层架构

为解决这一痛点，我们引入了 ACM MM 2025 国际顶会论文 **HM-RAG（Hierarchical Multi-agent Multimodal Retrieval-Augmented Generation）** 的核心架构思想，对现有系统进行重构。

HM-RAG 的核心理念是：**不把所有问题都丢给单一检索器，而是让一个"大脑"先做意图分解，再将不同子任务分发给专门的检索智能体**。我们将此思想精炼为三层智能体架构：

| 层级 | 角色 | 对应功能 |
|---|---|---|
| **第一层：分解智能体** | 意图路由器 | 分析用户问题，输出 `GRAPH` / `VECTOR` / `BOTH` 路由指令 |
| **第二层：多源检索智能体** | 并行检索 | Graph 分支走 Neo4j 结构化查询；Vector 分支走 ChromaDB 语义检索 |
| **第三层：决策智能体** | 融合回答 | 接收多源数据，去噪融合，生成最终解答 |

引入的 **ChromaDB** 作为纯本地向量数据库，与 Neo4j 形成互补：Neo4j 擅长结构化实体关联检索，ChromaDB 擅长非结构化文本的语义相似度检索。二者并行，形成真正的混合检索能力。

## 三、场景应用：语义搜索如何精准命中文化精神

以两个典型场景为例：

**场景一：宏观精神语义检索。** 当用户提问"什么是湖湘文化中'吃得苦、霸得蛮'的精神？"时，分解智能体会判定此问题属于宏观抽象语义，路由指令为 `VECTOR`。系统随即调用 ChromaDB 语义检索，将用户提问转化为向量，与预先入库的湖湘文化文献片段进行余弦相似度比对。ChromaDB 能够精准命中数据库中关于"霸蛮精神"、"吃得苦中苦，方为人上人"等相关文化片段，返回高相似度文本，而非图数据库中的空白结果。

**场景二：人物实体+精神内涵的复合查询。** 当用户提问"曾国藩的家书体现了怎样的湖湘精神？"时，分解智能体会判定需要同时检索人物实体信息（图谱分支）和精神内涵语义（向量分支），路由指令为 `BOTH`。Neo4j 负责返回曾国藩的生平、著作等结构化事实，ChromaDB 负责返回与"曾国藩家书"、"修身齐家"相关的语义片段。第三层决策智能体将两路数据融合，既给出有据可查的事实信息，又阐释了其中蕴含的湖湘精神。

## 四、工程价值：零显卡算力下的顶会架构落地

本方案最具推广价值的一点在于其**极致的工程性价比**：

- **LLM 引擎**：采用 MIMO 平台提供的 `mimo-v2.5-pro` 模型，通过云端 API 调用，本地无需部署大模型。
- **向量数据库**：ChromaDB 为纯 Python 实现的本地轻量级数据库，无需任何 GPU 算力，安装即用。
- **图数据库**：Neo4j 社区版免费，本地 Docker 部署即可运行。

整套架构无需任何昂贵显卡、无需任何付费云服务，仅凭一台普通笔记本电脑即可完整运行 ACM MM 顶会级别的多智能体混合检索系统。这种"零显卡、低成本、全本地"的部署模式，为中小团队乃至个人开发者复现前沿学术架构提供了极具参考价值的工程范式。
