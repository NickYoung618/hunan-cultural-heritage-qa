## 一、 今日解决的核心问题及根因剖析

在今天的多线程并发入库与图谱检索实测中，系统遭遇了三个隐蔽的工程化 Bug。现已全部通过白盒调试修复，具体复盘如下：

### 1. 长文本属性的“并发覆盖”与丢失 (Concurrency Overwrite)
* **表现症状**：大模型成功提取了如“气一元论”等长达百字的哲学解释，但在最终图谱中，该节点的 `description` 字段依然为空。
* **根因分析**：在 `agent_batch_run.py` 的多线程环境下，包含详尽解释的文本块（线程 A）与未包含解释的提及文本块（线程 B）几乎同时入库。由于原版 Cypher 采用了无条件覆盖指令 `SET n.description = $description`，导致后执行的“空字符串”无情洗刷了前一个线程的劳动成果。
* **修复方案**：将无条件 `SET` 重构为基于 `ON CREATE` 和 `ON MATCH` 的条件更新逻辑。使用 `CASE WHEN` 配合 `size()` 函数，强制数据库**只保留更长的文本**，拒绝空字符串覆盖。
* *注：排查中发现 Neo4j 5.x 严格区分了路径长度（`length()`）与字符串/列表大小（`size()`），误用 `length()` 会导致 `Type mismatch` 报错，现已统一纠正为 `size()`。*

### 2. 前端可视化呈现“节点大爆炸与文本错位” (UI Parsing Misalignment)
* **表现症状**：Web UI 绘制出的关系图极度混乱，出现名为“人物”的连线，以及名为一百多字长文（实际上是 description）的离谱节点。
* **根因分析**：前后端解析逻辑脱节。后端的 `rag_agent.py` 生成的 Cypher 为了保留长文本，已将返回值展平为 8 列（如 `a_name`, `relation`, `b_description` 等）。但前端 `web_ui.py` 仍在使用旧版的“按索引盲取”（取第0个、第1个、最后一个 key）逻辑，导致数据彻底错位。
* **修复方案**：重构 `web_ui.py` 中的 `build_evidence_graph` 函数，通过 `if "a_name" in record` 精准识别展平字典的 Key，显式提取 `a_type` 和 `b_type`，并加入了节点名称超长（>20字符）的安全屏蔽锁。

### 3. 多跳推理的“语义弥散”与幻觉 (Semantic Dispersion & Hairball)
* **表现症状**：查询“王夫之的儿子”时，大模型只答出一个儿子并说“三岁殇”（明显是错误的幻觉），且前端可视化呈现出由 35 条线构成的“海胆图”。
* **根因分析**：由于缺少关系词收束，大模型在抽取时自由发挥，造出了“四子”、“次子”、“生子”、“儿子”等同义词。为了防查不到，我们在检索层使用了泛匹配 `-[r]-`，结果将王夫之所有的朋友、老师、政敌等 36 个实体全部召回。庞杂的噪音数据（Token 溢出）直接冲垮了 LLM 的注意力机制，导致其在海量 JSON 中找不到真正的答案。
* **修复方案**：在 `graph_writer.py` 入库前植入**“动态关系规范化 (Normalization) 拦截器”**。通过定义 `RELATION_MAP` 词表，在 Cypher 执行前将所有的“四子、次子”强制收束为标准本体概念“父子”，从源头消灭了语义弥散。

## 二、 当前系统架构优缺点评估

经过今日的重构，系统已从一个“Demo”级别演进为具备初步生产能力的“工程级”知识图谱系统。

### 🌟 系统当前优点 (Pros)
1. **坚不可摧的幂等性与数据保护**：多线程 `MERGE` + 错峰限流（`MAX_WORKERS=3`）+ 基于 `size()` 的条件写入，确保了不论跑多少次爬虫，图谱数据只会越积越丰富，绝不会产生脏节点或丢失长文本。
2. **富属性图谱 (Property Graph) 能力**：成功将长达数千字的文言文评价与深奥的哲学释义挂载到边和节点上，真正实现了图谱存储与 LLM 理解的深度绑定。
3. **极速响应的检索体验**：基于双引擎（Sentence-transformers/N-gram）的语义缓存系统极为惊艳，相同问题拦截耗时仅 **0.03 秒**，大幅节省 API Token 成本。
4. **前端演示表现力极强**：暗色风格的 Gradio 6.x 结合 Plotly 交互式绘图，数据呈现专业、直观，极具项目申报与科研汇报的视觉冲击力。

### ⚠️ 系统当前缺点与瓶颈 (Cons)
1. **静态本体词表存在维护成本**：当前的 `RELATION_MAP` 仍是写死在代码里的字典。若后续引入大量新语料，可能需要频繁人工修改映射表。
2. **检索策略略显保守**：为了对抗此前的关系词混乱，`rag_agent.py` 中的【绝对强制法则零】依然禁止大模型在 Cypher 中指定具体的连线类型（强制用 `-[r]-`），这在当前关系已标准化的情况下，反而成了限制精准检索的阻碍。
3. **多跳路径的可视化局限**：目前对于二度以上的关系网络，前端依然是平铺展示，尚未针对复杂的时间线或强弱关系进行加权视觉渲染。

## 三、 下一步开发任务 (Next Steps)

针对上述评估，V2.4 版本的核心演进路线已明确。接下来的行动将聚焦于**“释放精确检索能力”**与**“应用层扩展”**：

* **任务 A：解绑检索限制，重写 Text-to-Cypher Prompt**
  * **行动**：修改 `rag_agent.py`，**废除旧版的【法则零】**。
  * **新策略**：在 Prompt 中明确提供系统现有的标准化本体词表（如：“已知亲属关系为：父子、夫妻、兄弟”），允许甚至鼓励快模型在生成 Cypher 时使用类型约束（如 `MATCH (a)-[r:父子|兄弟*1..2]-(b)`），从“捞网式泛匹配”升级为“狙击式精确匹配”，彻底告别冗余数据。

* **任务 B：本体字典的工程化外置**
  * **行动**：将 `RELATION_MAP` 从 `graph_writer.py` 剥离，抽离成一个独立的 `ontology_config.json` 或 `.yaml` 配置文件，方便未来无需修改代码即可动态扩充湖湘文化的专业知识类目。

* **任务 C：图谱分析 (Graph Analytics) 与 UI 深度整合**
  * **行动**：目前 `graph_analytics.py` 已经具备计算 PageRank 和 最短路径的能力，接下来需探索在 Web UI 问答中，当触发核心历史人物时，自动在侧边栏显示“图谱核心枢纽指数（PageRank 分数）”，增强系统的学术厚度。
