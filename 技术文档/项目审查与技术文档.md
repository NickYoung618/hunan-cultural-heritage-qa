## 一、项目概述与申报目标对照

### 1.1 项目定位

本项目是一个基于 **LangGraph + MiMo (小米大模型) + Neo4j** 的 GraphRAG（图谱增强生成）多智能体问答系统，服务于**湖湘文化遗产的数字化传承**。核心研究对象包括王夫之（船山先生）、曾国藩、周敦颐等湖湘历史名人。

### 1.2 申报目标分析（基于项目技术文档推断）

申报项目应聚焦以下核心创新点：

| 申报要点 | 当前实现状态 | 完成度 |
|---------|-------------|--------|
| 基于知识图谱的文化遗产数字化 | ✅ Neo4j 图数据库已部署，实体/关系已入库 | 85% |
| 多智能体协作的文本抽取 | ✅ LangGraph DAG + Extractor/Reviewer/Writer 三智能体 | 90% |
| GraphRAG 问答引擎 | ✅ Text-to-Cypher + 证据检索 + LLM 总结 | 80% |
| 图谱分析与隐性知识挖掘 | ✅ PageRank/最短路径/介数中心性已实现 | 75% |
| 语义缓存与性能优化 | ✅ 双引擎语义缓存（sentence-transformers/n-gram） | 85% |
| 可视化交互界面 | ✅ Gradio 6.x + Plotly 交互式关系图 | 80% |
| 本体标准化与关系规范化 | ⚠️ 已实现但硬编码在代码中，未外置配置 | 60% |
| 规模化数据采集 | ⚠️ 爬虫已实现但覆盖人物有限 | 50% |

### 2.2 核心模块逐一审查

#### 2.2.1 实体抽取引擎 (	ools/extractor.py) — 评分: ⭐⭐⭐⭐

**已实现的技术亮点：**
- ✅ Pydantic v2 Schema 严格定义（Entity/Relationship/GraphExtractionResult）
- ✅ DeepSeek JSON Mode 强制结构化输出
- ✅ 多重校验 + 优雅降级（部分恢复机制）
- ✅ 最多重试 2 次，失败返回空结果而非崩溃
- ✅ description/detail 字段的原文摘抄能力

**现存问题：**
- ⚠️ System Prompt 中的 JSON_SCHEMA_DEFINITION 示例只覆盖了 气一元论理势合一等少数概念，对其他类型（事件、官职、学派）的示范不足
- ⚠️ max_length=50 对 Entity.name 的物理截断可能导致合法长名称被截断（如某些古籍名称超过15字）
- ⚠️ 噪音过滤规则仅针对影视剧演员，未覆盖现代纪念活动等无关内容

#### 2.2.2 图谱入库引擎 (	ools/graph_writer.py) — 评分: ⭐⭐⭐⭐

**已实现的技术亮点：**
- ✅ MERGE 幂等写入，无惧重复运行
- ✅ ON CREATE / ON MATCH 条件更新，保护长文本不被空字符串覆盖
- ✅ size() 函数正确替代 length()（Neo4j 5.x 兼容）
- ✅ 实体与关系分别写入，事务安全性良好

**现存问题：**
- ⚠️ 关系类型未标准化（elation.replace(' ', '_') 过于简单，如四子仍会成为独立关系类型）
- ⚠️ 缺少批量写入优化（逐条执行 Cypher，未使用 UNWIND 批量操作）
- ⚠️ 未实现关系规范化拦截器（RELATION_MAP 仅在技术文档中提及，代码中未落地）

#### 2.2.3 批量入库调度器 (gent_batch_run.py) — 评分: ⭐⭐⭐⭐

**已实现的技术亮点：**
- ✅ LangGraph StateGraph 编排三智能体流水线
- ✅ ThreadPoolExecutor 多线程并发（MAX_WORKERS=3）
- ✅ 错峰发车（0.5s 间隔）+ 局部退避（10s 冷却）
- ✅ 长文本切片（600字/块）+ 批量目录读取
- ✅ 单块失败不影响整体流程

**现存问题：**
- ⚠️ 线程池中 reviewer_node 抛出 ValueError 会导致整个线程异常终止
- ⚠️ 缺少进度持久化（中断后无法从断点恢复）
- ⚠️ 未实现抽取结果的 JSON 缓存（data/json_cache/ 目录已建但未使用）

#### 2.2.4 问答中枢 (ag_agent.py) — 评分: ⭐⭐⭐⭐

**已实现的技术亮点：**
- ✅ 流式输出 + 大小模型路由
- ✅ 语义缓存（0.03s 极速命中）
- ✅ Cypher 生成 → 执行 → 证据序列化 → 回答生成完整链路
- ✅ rich 终端美化 + 特殊指令（shortest path, /report）
- ✅ _serialize_evidence 深度属性提取

**现存问题：**
- 🔴 Cypher Prompt 中的「法则零」（禁止硬编码关系类型）在关系已标准化后反而成了阻碍，应允许精确匹配
- ⚠️ fast_model 和 pro_model 配置为同一个模型（mimo-v2.5-pro），未实现真正的大小模型分离
- ⚠️ 缺少回答溯源能力（无法标注答案中哪些部分来自哪条图谱证据）

#### 2.2.5 图谱分析引擎 (	ools/graph_analytics.py) — 评分: ⭐⭐⭐⭐

**已实现的技术亮点：**
- ✅ GDS 库自动检测 + 纯 Cypher 回退
- ✅ shortest_path / all_shortest_paths + 模糊匹配兜底
- ✅ degree_centrality / pagerank / betweenness_centrality
- ✅ full_report 综合报告生成

**现存问题：**
- ⚠️ PageRank 的纯 Cypher 回退使用加权度中心性近似，精度有限

#### 2.2.6 Web 界面 (web_ui.py) — 评分: ⭐⭐⭐⭐

**已实现的技术亮点：**
- ✅ Gemini 暗色主题，视觉专业度极高

- ✅ build_evidence_graph 精准展平字典解析

**现存问题：**
- ⚠️ 节点超长名称（>20字符）直接屏蔽，可能丢失有效信息
- ⚠️ 缺少图谱导出功能（SVG/PNG）

#### 2.2.7 语义缓存 (cache_tool/semantic_cache.py) — 评分: ⭐⭐⭐⭐⭐

### 3.2 关键工程决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 图数据库写入语法 | MERGE 而非 CREATE | 幂等性，防重复 |
| 长文本保护 | ON MATCH + size() 比较 | 只保留更长版本 |
| 并发控制 | ThreadPoolExecutor(3) | 平衡速度与 API 限流 |
| 实体抽取输出 | Pydantic + JSON Mode | 替代脆弱的字符串清洗 |
| 缓存策略 | 语义相似度匹配 | 替代精确字符串匹配 |
| 前端框架 | Gradio 6.x | 快速原型，适合科研演示 |

## 四、待实现技术细节与改进方案

### 4.1 🔴 高优先级

#### 4.1.1 本体配置外置化

**现状**：关系规范化映射（RELATION_MAP）写死在代码或仅存在于技术文档设计中。

**方案**：
`json
// ontology_config.json
{
  \entity_types\: [\人物\, \地点\, \作品\, \事件\, \概念/流派\, \时间\, \学派\, \官职\],
  \relation_normalization\: {
    \四子\: \父子\, \次子\: \父子\, \生子\: \父子\, \长子\: \父子\,
    \妻子\: \夫妻\, \夫人\: \夫妻\, \配偶\: \夫妻\,
    \兄长\: \兄弟\, \弟弟\: \兄弟\,
    \学生\: \师生\, \弟子\: \师生\, \门生\: \师生\,
    \老师\: \师生\, \恩师\: \师生\,
    \撰写\: \著作\, \编写\: \著作\, \著有\: \著作\,
    \隐居\: \隐居于\, \居住\: \隐居于\
  },
  \stop_relation_patterns\: [\相关\, \有关\, \联系\, \涉及\]
}
`

**改动文件**：
- 新建 ontology_config.json
- 修改 	ools/graph_writer.py：入库前加载配置并执行规范化
- 修改 ag_agent.py：在 Cypher Prompt 中注入标准化本体词表

#### 4.1.2 解绑 Cypher 检索限制

**现状**：法则零强制使用 -[r]- 泛匹配，导致冗余数据召回。

**方案**：废除法则零，在 Prompt 中提供标准关系类型列表，鼓励精确匹配：
`
已知标准关系类型：父子、夫妻、兄弟、师生、著作、评价、隐居于、出生于、核心思想、流派传承。
请优先使用类型约束：MATCH (a)-[r:父子|师生*1..2]-(b)
`

#### 4.1.3 批量写入性能优化

**现状**：逐条执行 Cypher INSERT，N 个实体需要 N 次网络往返。

**方案**：使用 UNWIND 批量操作：
`cypher
UNWIND  AS e
MERGE (n:\{e.type}\ {name: e.name})
ON CREATE SET n.description = e.description
ON MATCH SET n.description = CASE
  WHEN e.description <> \\ AND (n.description IS NULL OR size(e.description) > size(n.description))
  THEN e.description ELSE n.description END
`

**预期收益**：入库速度提升 5-10 倍。

### 4.2 🟡 中优先级

#### 4.2.1 JSON 缓存启用

**现状**：data/json_cache/ 目录已建但未使用。

**方案**：在 extractor.py 的 extract() 方法中，以文本 hash 为 key 缓存抽取结果，避免重复调用 LLM API。

#### 4.2.2 回答溯源能力

**现状**：用户无法知道答案中的每个事实来自哪条图谱边。

**方案**：在 _generate_answer 的 Prompt 中要求 LLM 为每个事实标注来源，前端以角标形式展示。

#### 4.2.3 数据采集扩展

**现状**：仅覆盖王夫之等少数人物。

**方案**：扩展爬虫名单，建议分批覆盖：
- **第一批（核心人物）**：王夫之、王介之、曾国藩、左宗棠、胡林翼、彭玉麟
- **第二批（学术传承）**：周敦颐、张栻、朱熹、王阳明
- **第三批（湖湘学派）**：魏源、谭嗣同、黄兴、蔡锷

#### 4.2.4 大小模型路由真正实现

**现状**：ast_model 和 pro_model 都指向 mimo-v2.5-pro。

**方案**：
- Cypher 生成使用轻量快速模型（降低延迟）
- 回答撰写使用高质量模型（提升答案质量）

### 4.3 🟢 低优先级

#### 4.3.1 图谱版本管理
- 实现图谱快照导出/导入
- 支持多版本图谱对比

#### 4.3.2 用户反馈闭环
- 添加回答质量评分（👍/👎）
- 低分回答自动触发重新生成或检索

#### 4.3.3 多模态支持
- 接入 OCR 识别古籍扫描件
- 支持图片中的实体抽取
