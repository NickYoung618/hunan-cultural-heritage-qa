# 面向湖湘文化遗产数字化传承的 GraphRAG 系统
**—— 全链路开发与工程避坑实录**

## 一、 环境基座与配置层

### 1.1 `.env` (全局环境变量配置)
* **构建逻辑**：采用前后端解耦的安全设计，将大模型（DeepSeek API、Base URL、Model Name）和图数据库（Neo4j URI、用户名、密码）的敏感凭证统一存储在此文件中，避免硬编码。
* **遭遇问题**：在启动 V2 版本多线程数据库（8687 端口）时，数据依然被错误地写入了初代数据库（7687 端口）。
* **解决方案**：排查发现 `.env` 文件中缺失了 Neo4j 的环境配置，导致 Python 脚本回退触发了代码中预设的默认值（7687）。通过在 `.env` 中补全 `NEO4J_URI="bolt://localhost:8687"` 等三行配置，并在代码中同步修改备用端口，彻底实现了新旧数据库的物理隔离。

### 1.2 `docker code.txt` (容器化部署配置)
* **构建逻辑**：记录了使用 Docker 部署 Neo4j 图数据库的完整生命周期指令（`run`, `start`, `stop`, `ps`, `logs`）。通过容器化技术，实现了项目数据的物理隔离与一键启停。
* **遭遇问题**：在 Windows CMD 中直接运行带有换行符 `\` 的多行 Docker 创建命令时，终端直接报错并拒绝执行。
* **解决方案**：系统底层语法差异所致。Windows CMD 不识别 Linux 的 `\` 换行符。将其替换为脱字符 `^`，或直接将其合并为单行长命令后，容器成功启动。

## 二、 数据采集与处理层

### 2.1 `data/` 目录与 `tools/spider_baike.py` (语料采集中心)
* **构建逻辑**：为确保系统具备高密度的知识源，通过 Claude Code 辅助编写了针对湖湘文化代表人物的定向爬虫脚本（`spider_baike.py`）。
* **工程产出**：抓取并清洗出诸如 `wangfuzhi_baike.txt` 等数万字的纯文本语料，存储于 `data/` 目录下，作为后续大模型进行知识提取的原始“矿石”。

### 2.2 `tools/extractor.py` (大模型知识抽取器)
* **构建逻辑**：这是图谱构建的“大脑”。封装了 DeepSeek API，通过精心设计的 Prompt，引导大模型阅读古文，并严格按照预设的 Schema（人物、作品、地点、事件等）抽取实体，以及实体间的关系，最终返回结构化的 JSON 数据。
* **遭遇问题**：在处理大段落复杂古文时，频繁抛出 `Request timed out`（请求超时）报错，导致 LangGraph 节点直接崩溃。
* **解决方案**：大模型进行深度语义分析并生成严格格式的 JSON 极其消耗算力，默认的 30 秒等待时间过短。通过修改 OpenAI Client 的初始化参数，将 `timeout` 大幅放宽至 `120.0` 甚至 `300.0` 秒，赋予了大模型充足的“思考时间”，彻底解决了断连问题。

### 2.3 `tools/graph_writer.py` (图谱入库连接器)
* **构建逻辑**：封装了 Neo4j 的 Python Bolt 驱动。负责将 `extractor.py` 吐出的 JSON 文本转化为 Cypher 语言，并物理写入数据库。
* **遭遇问题**：大模型输出的内容偶尔会包含 Markdown 标记（如 ````json ````），导致 Python 原生 `json.loads()` 解析失败。同时，多次运行脚本可能导致节点重复创建。
* **解决方案**：
  1. 增加了字符串容错清洗逻辑（`.strip()` 剥离 Markdown 标记）。
  2. 放弃使用 `CREATE`，全面改用 Cypher 的 `MERGE` 语法，赋予了系统数据库级别的“幂等性”，实现了无惧重复运行的稳健入库。

## 三、 多智能体协同与调度层

### 3.1 `agent_workflow.py` & `agent_workflow_v2.py` (工作流编排)
* **构建逻辑**：利用 LangGraph 构建了核心的有向无环图（DAG）流水线。定义了 `HuxiangState` 作为状态托盘，将工作分解为三个专职智能体：提取专家（Extractor）、审核专家（Reviewer）和入库专家（Writer）。数据只能单向流转，确保了加工逻辑的严密性。

### 3.2 `agent_batch_run.py` (高并发批量处理引擎)
* **构建逻辑**：为了将几万字的长文灌入图谱，设计了此批量引擎。它包含一个长文本切片机（按自然段切割为 600 字的安全数据块），并循环调用上述的 LangGraph 工作流。
* **遭遇问题 1 (单线程限流)**：大批量高频请求大模型时，被 API 服务端拦截，返回 HTTP 429 (Too Many Requests) 错误，导致审核节点频繁驳回数据。
* **解决方案 1**：加入初步的防抖动机制，在每次循环末尾强制 `time.sleep(5)` 进行冷却。
* **遭遇问题 2 (性能瓶颈)**：单线程外加休眠机制导致数万字的入库极其缓慢，宛如“拖拉机”，无法支撑后续百余位湖湘名人的规模化扩展。
* **解决方案 2 (终极满血版架构)**：
  1. 引入 `concurrent.futures.ThreadPoolExecutor`，重构为**多线程并发架构**。
  2. 设置 `MAX_WORKERS = 3` 作为并发安全阀，贴合 API 限流边缘。
  3. 引入**局部退避机制（Exponential Backoff 雏形）**：当某个线程遇到抽取失败或超时时，仅该线程执行 `time.sleep(10)`，其他线程继续保持全速运转。
  4. 引入**错峰发车** (`time.sleep(0.5)`)，平滑发包，彻底解决了高并发与 API 速率限制之间的矛盾。

## 四、 运维可视化与应用层

### 4.1 `clean_db.py` (一键重置工具)
* **构建逻辑**：在反复调试系统期间，需要经常清理脏数据。通过封装 `MATCH (n) DETACH DELETE n` 的终极清仓咒语，实现了一键将图数据库恢复为空白状态的运维脚本。

### 4.2 Neo4j Browser (可视化调试面板)
* **遭遇问题**：在浏览器中打开 `http://localhost:8484` 时，界面布局极度丑陋错乱，且输入的 Cypher 代码发生语法错误（如变成了“匹配 (n)-[r]->(m) 返回”）。
* **解决方案**：排查确认为浏览器自带的“网页自动翻译”功能强行将英文前端面板的控制代码和 CSS 样式翻译成了中文。通过在浏览器地址栏右侧禁用该网站的机器翻译并刷新，成功恢复了高级极客感的节点网络可视化面板。

### 4.3 `rag_agent.py` & `main.py` (图谱问答中枢)
* **构建逻辑**：系统最终的价值展现点。实现了真正的 GraphRAG 闭环。分三步执行：第一步，LLM 将自然语言意图翻译为 Cypher 语句；第二步，Python 将语句发往 Neo4j 提取物理连线作为确凿证据；第三步，LLM 依托证据生成学术总结。
* **遭遇问题**：当询问复合问题（“王夫之和王介之是什么关系？他有哪些著作？”）时，Neo4j 报错 `All sub queries in an UNION must have the same return column names`。大模型为了拼凑两次查询的结果，违规使用了 `UNION ALL`。
* **解决方案**：运用 Prompt Engineering（提示词工程），在 `cypher_prompt` 中加入高优级的强制约束指令：`⚠️【极其重要】：严禁使用 UNION 或 UNION ALL。遇到复合问题，请使用多条 MATCH 语句，并统一在最后一行 RETURN 所有的结果。` 成功规避了该语法陷阱，赋予了系统完美的“多跳逻辑推理”能力。
