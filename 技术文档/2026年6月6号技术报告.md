## 二、检索架构升级：GraphRAGExpert 重构

### 2.1 背景

原 `rag_agent.py` 实现的是 HM-RAG 三层混合检索的测试版本（仅 2 条硬编码测试数据），缺少 `GraphRAGExpert` 类，导致 `web_ui.py` 无法运行。

### 2.2 重构方案

重写 `rag_agent.py`，实现完整的 `GraphRAGExpert` 类，融合 Neo4j 图谱检索 + ChromaDB 向量检索。

**类结构**：

```python
class GraphRAGExpert:
    ├── graph: GraphRetriever          # Neo4j 图谱检索器
    ├── vector: VectorRetriever        # ChromaDB 向量检索器
    ├── analytics: _AnalyticsProxy     # 图谱分析代理
    │
    ├── ask_stream(question)           # 流式问答（核心方法）
    ├── ask(question)                  # 非流式问答
    ├── query_shortest_path(n1, n2)    # 最短路径查询
    └── _filter_relevant_evidence()    # LLM 智能证据筛选
```

### 2.3 流式问答流程

```
用户提问
    ↓
第一步：图谱检索（Neo4j）
    ├── 提取问题中的已知实体（如"曾国藩"）
    ├── 查询该实体的全部关系（无 LIMIT 限制）
    └── 返回原始关系列表
    ↓
第二步：LLM 智能筛选（核心创新）
    ├── 将全部关系格式化为编号列表
    ├── 发送给 MIMO LLM，要求根据问题语义筛选相关条目
    ├── LLM 返回相关条目的编号
    └── 过滤出相关证据
    ↓
第三步：向量检索（ChromaDB）
    ├── 将问题作为查询文本
    └── 返回 top-5 语义最相关的文档块
    ↓
第四步：证据融合 + LLM 流式生成
    ├── 将图谱证据、向量证据格式化为文本
    ├── 拼接成完整的 prompt
    └── MIMO LLM 流式生成最终回答
    ↓
输出：answer + cypher + evidence_graph + stats
```

### 2.4 LLM 智能证据筛选（核心创新）

**问题**：图谱查询返回实体的全部关系（如左宗棠有 174 条关系），其中大量与问题无关（如问"成就"时返回了家族关系）。

**旧方案（已废弃）**：硬编码关键词映射表
```python
INTENT_FILTER_MAP = {
    "著作": ["著有", "撰写"],
    "家人": ["父子", "兄弟"],
    ...
}
```
缺点：每种问法都需要手动维护，无法覆盖所有场景。

**新方案**：让 LLM 自己判断哪些关系与问题相关

```python
def _filter_relevant_evidence(self, question, evidence):
    # 将全部关系格式化为编号列表
    evidence_lines = [f"[{i}] {src} --[{rel}]--> {tgt}" ...]

    # 发送给 LLM 筛选
    filter_prompt = f"""用户的问题是："{question}"
    请筛选出与问题直接相关的关系，只返回编号。"""

    # LLM 返回：0,1,3,5
    # 过滤出对应条目
```

**实测效果**（以左宗棠为例，原始 174 条关系）：

| 问题 | 筛选后条数 | 筛选结果 |
|------|-----------|---------|
| 左宗棠的成就 | 51 条 | 收复、创建、领导、参与... |
| 左宗棠的家人 | 21 条 | 父子、兄弟、夫妻、父女... |
| 左宗棠的思想 | 23 条 | 主张、批评、受思想影响... |

**优势**：不管用什么角度提问，LLM 都能理解语义并正确筛选，无需维护任何映射表。

### 2.5 Neo4j 查询优化：移除 LIMIT 限制

**问题**：原查询有 `LIMIT 20`，当实体关系超过 20 条时会被截断。

**修复**：移除 `query_entity_relations()` 和 `query_keyword()` 中的 `LIMIT` 限制，改为查询全部关系，由 LLM 在后续步骤中智能筛选。

**效果**：左宗棠从 20 条 → 174 条完整关系，LLM 再从 174 条中筛选出与问题相关的子集。

### 2.6 web_ui.py 兼容性修复

**问题 1**：`build_evidence_graph()` 只认 `a_name`/`b_name` 格式，但图谱查询返回的是 `source`/`target` 格式，导致可视化为空。

**修复**：在 `build_evidence_graph()` 中增加对 `source`/`target` 格式的兼容处理。

**问题 2**：Cypher 展示区显示的是占位符 `WHERE a.name CONTAINS '...'`，不是实际执行的查询。

**修复**：改为显示真实的 Cypher 查询语句，包含实际的实体名称和 LLM 筛选信息。

## 四、网络检索模块上线（Tavily API）

### 4.1 背景

HM-RAG 论文的三层混合检索架构包含第三路——网络检索（Web Search）。原有系统只有图谱 + 向量两路检索，对于本地知识库未覆盖的内容（如抽象概念"湖湘精神"），LLM 会使用自身训练知识生成回答，存在幻觉风险。

### 4.2 实现方案

采用 **Tavily Search API**（AI 优化的搜索引擎），通过 `tavily-python` SDK 调用。

**申请流程**：
1. 访问 https://tavily.com 注册账号
2. 在 Dashboard 获取 API Key
3. 配置到 `.env` 文件：`TAVILY_API_KEY=你的密钥`
4. 免费版每月 1000 次搜索

**代码实现**：`WebRetriever` 类

```python
class WebRetriever:
    """Tavily 网络检索器。"""
    def query(self, question, num_results=5):
        response = self._client.search(
            query=question,
            max_results=num_results,
            search_depth="advanced",
            include_answer=True,
        )
        # 返回 answer + 搜索结果列表
```

**回退机制**：Tavily API 失败时，自动回退到本地已爬取的百科数据检索。

## 五、性能优化：证据筛选从 5 分钟降到 0.05 秒

### 5.1 问题

初始实现中，LLM 智能筛选需要将实体的全部关系（如左宗棠 174 条）发送给 MIMO API 进行筛选，耗时超过 5 分钟。

### 5.2 优化方案

采用**关键词评分 + 预过滤**替代 LLM 筛选：

```python
# 1. 预过滤：去重 + 按关系类型分类（高价值/低价值）
# 2. 关键词评分：根据问题意图给每条关系打分
# 3. 按分数排序，取 top-20
```

**评分逻辑**：
- 关系类型包含问题关键词：+5 分
- 高价值关系（著有、创建、领导等）：+3 分
- 低价值关系（家族关系等）：-2 分
- 家族查询时反转评分权重

### 5.3 优化效果

| 问题 | 优化前 | 优化后 |
|------|--------|--------|
| 曾国藩的成就 | 5+ 分钟 | 0.05 秒 |
| 左宗棠的思想 | 5+ 分钟 | 0.02 秒 |
| 曾国藩的家人 | 5+ 分钟 | 0.03 秒 |

## 七、遗留问题与后续规划

### 7.1 遗留问题

1. **短文件质量**：岳麓书院、湘军、经世致用三个词条的爬取内容较短（百度百科碎片），信息密度有限。可考虑后续补充更多数据源。
2. **LLM 筛选延迟**：智能证据筛选需要额外调用一次 LLM API，增加了约 1-2 秒的响应时间。

### 7.2 后续规划

1. **优化 LLM 筛选性能**：可考虑将筛选逻辑改为并行执行，或使用更轻量的模型进行筛选。
2. **扩展数据源**：为岳麓书院、湘军等短词条补充更多爬取来源（如学术论文、地方志等）。
3. **多轮对话支持**：当前系统为单轮问答，后续可加入对话历史管理。
